Project Description

Olin Organizer (O2) is a timetable builder and course registration planner for Olin College students. It uses python and tkinter library.

The Big Idea

To create a schedule planner for Olin students to configure their timetable

Minimum Viable Product

  • The minimum viable product is to allow Olin students to plan whichever classes they are planning to enrol in/ already enrolled into a calendar in the website that we have created. They will be able to check if the classes clashes with another class and easily plan for possible timetables for course registration.

Stretch Goal

  • To export data as ical/google calendar, and calendar as JPEG.PDF file.
  • To convert into a webpage which can fit into the resolution of any smartphone nicely.

Learning Goals

Shared Goal

  • We would like to become better in pair programming as we think it will be beneficial to us in the future.
  • We would also like to learn how to do web programming.

Individual Goal

  • We feel that one of the main individual learning goal should be to learn how to program from one another.

Implementation Plan

We will be using HTML5, CSS, JavaScript, Python.
Library we will be using: jQuery & Flask
We plan to extract course information from Olin, Babson and Wellesley college websites and convert them into data format which can be eaily translated onto calendar.

Project Schedule

Week 1

  • Extract classes information online from Olin, Babson and Wellesley college

Week 2

  • Import and format the information that we have extracted into Python
  • Read and study about all the libraries we are using and try out different possible way of drawing
  • Get used to web programming

Week 3

  • Create UML diagram and divide parts
  • Create the class needed for drawing the background of the calendar
  • Create the class that stores all the information of the classes available in Olin
  • Generate code for interactive programming

Week 4

  • Continue the task from week 3
  • Start integrating the codes
  • Test out several different scenarios of overlapping schedules/exams
  • Create a portal/comment box for past students to add their review about courses

Week 5

  • Testing of program/website
  • Look for improvements or beautifying layout of program/website

Collaboration Plan

  • We plan to split different tasks at the beginning of the project so that acquiring information and learning process can be much faster.
  • Pair programming will be done throughout the whole program so that every member will be keeping up at the same level.

Risks

We have to learn web programming all on our own and look for graphical libraries

Design Review

Preparation and Framing

We hope that other groups gets to know about what our project is about. Therefore, we plan to start the review by giving an overview on Olin Organizer. In addition, it would be easy for the audience to participate because this project could be a potential program in Olin. We want to find out what functions does students want from Olin organizer. There are too many functions that we can implement in a program so we need feedback to sieve out those that many will prefer. We will provide a agenda for the design review so our audience will know the flow of presentation. We also plan to get audience involved by allocating a time for them to give their comments.

Presentation Flow

  1. Overview of our project: Project Description
  2. Agenda of presentation
  3. First draft of our product GUI
  4. Features
    • Explain features we are planning on implementing
    • Explain how the features interact with one another
    • Get feedback on the features they would like to see/ they do not require
  5. Libraries
    • Libraries we are intending to use
    • Feedback on additional libraries we could look into/ on the libraries we are going to use
  6. Timeline: Steps we are planning to take
  7. Open feedback session

Code Review

Preparation and Framing

We plan to tell the audience that we are changing from website to program and we are using Model­View­Controller (MVC) coding structure. We are also going to explain how Model, View and Controller classes implemented in our code and how they interact with one another. There are several structure that a group can use in order to implement a function. As we are using MVC structure, we want to get feedback from the audience as well as Professor Benjamin. It could be a piece of advice on using MVC structure or giving us another option that could be better.

We will present our review as follow:

  1. Explain parts of our code – This will help audience understand how the program flows, we will also provide some graphical resources such as a prototype of GUI
  2. Give a brief description of code structure – We will explain some classes which are related to our key questions
  3. Release the key questions – Ensure the audience understands the reason of the key question as we will describe our current situation
  4. Receive feedback from audience – We will flash our code and ask for comments from the audience.

Questions:

  1. How can two (or more) different classes interact with one another - can inheritance work both ways?
  2. Is there a more effective way of implementing course search?
  3. Which widgets we should use from tkinter library?

June

Our first question was about how we should implement the search function in CourseList(). Before the code review, we planned to implement the search course function by looking through the entire course list to find the courses that matched given conditions. During the code review, most of audience agreed with our suggestion and hence, we decided to go on with this decision.

Our second question was about how two classes might interact both ways. In the previous project, some of us implement the interaction between two classes by setting one class as an attribute of the other class and access the class by calling its attribute. However, this method is only possible for one-way interaction. Since we have two classes that needs to have interaction with each other in parallel, we needed to find a new way to implement this. Fortunately, Prof. Ben gave a great suggestion about this problem. His suggestion was implementing interaction in another class that can access both classes. So we decided to follow his suggestion.

At first, we thought code review was a platform we could discuss our ways to implement code. Therefore, we wanted to discuss how we can implement some functions in our code such as searching method and handling interaction between classes. However, we didn’t provide enough context to the audience to be able to understand our code and our code had not been implemented much. As a result, we didn’t have enough question to ask and failed to make audience to be in the same page with us.

Jee Hyun

In general, the code review felt like a second design review. Some groups have gotten pretty far with coding. However, the review session was spent mostly on getting feedback on the design, function and layout we wanted to see more than how the code was being implemented currently and what were possible improvements that could be made to make it more efficient or professional. This might be due to the fact that going through the code is not an effective or engaging presentation method and the knowledge each group picks up doing our own project is not the same. Pre readings with questions and some background context of the questions might have led to a more useful code review session where we could discuss possible implementation plan in class. Also, UML diagram would have helped greatly when discussing the code.

Learning points from the code review:

  1. Frame the problems we want to ask and breakdown into simple questions
  2. If the question needs background context, make sure to prepare a pre reading so that we can have constructive discussion in class
  3. Have a UML diagram
